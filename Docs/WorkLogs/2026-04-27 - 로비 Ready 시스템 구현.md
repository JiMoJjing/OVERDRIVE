## Networking Impact

- Ready 상태는 PlayerState의 Replicated State로 관리합니다.
- 클라이언트는 Ready 값을 직접 확정하지 않고, PlayerController의 Server RPC를 통해 서버에 요청합니다.
- Late Join 클라이언트도 PlayerState Replication을 통해 현재 Ready 상태를 받을 수 있습니다.
- 이번 단계의 RPC 검증은 PlayerState 유효성 확인까지이며, 로비 상태별 허용 여부는 Lobby GameMode 단계에서 처리할 예정입니다.
- 로비 전체 요약 상태는 GameState의 Replicated State로 관리하며, 클라이언트 UI는 `OnLobbySummaryChanged` 델리게이트로 갱신할 수 있습니다.
- GameMode는 서버에서 PlayerArray를 순회해 전체 인원과 Ready 인원을 계산하고, 클라이언트 표시용 결과만 GameState에 기록합니다.
- Dedicated Server 기준으로는 `PostLogin`/`Logout`이 접속/퇴장 집계에 충분하지만, Listen Server와 PIE 테스트 안정성을 위해 `BeginPlay`에서도 초기 요약을 한 번 갱신합니다.
- Widget은 클라이언트 표시 계층이며 Ready 확정이나 GameState 변경을 직접 수행하지 않습니다.
- `Lobby.umap`은 로비 레벨로 사용하며, 맵 기본 GameMode는 `AOverdriveLobbyGameMode`로 설정합니다.
- 로비 UI 자동 표시는 테스트용 연결이며, 이후 로비/미션 전환 시점에 맞춰 표시/제거 흐름을 분리할 예정입니다.

## Decisions / Solutions

- `Lobby/`가 아니라 `Player/` 폴더에 배치했습니다. 이 PlayerState는 로비 전용이 아니라 미션/결과 화면까지 이어지는 플레이어 단위 상태의 기반 클래스이기 때문입니다.
- 지속 상태 동기화는 Multicast RPC가 아니라 Replicated Property로 처리했습니다.
- 클래스 내부 선언은 함수 선언을 먼저 모으고, 변수 선언을 뒤에 배치하는 방식으로 정리하기로 했습니다.
- 함수 선언, 함수 호출, Delegate 선언, 단순 조건식은 한 줄로 읽을 수 있으면 임의 줄바꿈하지 않는 방식으로 정리하기로 했습니다.
- WBP 에셋은 Editor Python으로 생성하되, 테스트 레이아웃과 버튼 동작은 `UOverdriveLobbyWidget` C++ 부모가 런타임에 구성하도록 했습니다.
- `Tools/CreateLobbyTestAssets.py`는 Subsystem 기반 API로 정리했고, Commandlet 실행 결과 `0 error / 0 warning`을 확인했습니다.
- 자동화 스크립트에는 UE 5.7 Python Commandlet의 Widget Designer 계층 편집 한계를 주석으로 남겼습니다.
