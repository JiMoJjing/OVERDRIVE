# Codex Workflow Rules

## 기본 원칙

Codex는 코드를 많이 작성하는 도구가 아니라, 프로젝트 규칙 안에서 통제된 변경을 수행하는 도구로 사용합니다.

항상 다음을 우선합니다.

1. 작은 작업 단위
2. 명확한 변경 이유
3. 서버 / 클라이언트 책임 구분
4. GAS / Replication 영향 확인
5. 작업 기록 가능성

---

## 작업 전 확인 사항

Codex는 작업 전에 다음을 확인합니다.

- 작업 목표
- 수정 예상 파일
- 변경 범위
- 관련 시스템
- 서버 권한 관련 여부
- GAS 관련 여부
- Replication 관련 여부
- 에셋 또는 에디터 설정 필요 여부

---

## 작업 중 규칙

- 요청된 작업에 집중합니다.
- 관련 없는 리팩토링을 하지 않습니다.
- 아키텍처를 조용히 바꾸지 않습니다.
- 한 번에 너무 많은 파일을 수정하지 않습니다.
- 큰 시스템을 통째로 재작성하지 않습니다.
- 새 플러그인이나 외부 의존성을 임의로 추가하지 않습니다.
- 설명 가능한 코드를 작성합니다.
- 임시 코드는 TODO와 함께 목적을 남깁니다.

---

## 작업 후 보고 규칙

작업 완료 후 다음 형식으로 보고합니다.

```text
Summary
- 변경 요약

Files Changed
- 파일 경로: 변경 이유

Technical Notes
- Networking 영향
- GAS 영향
- Unreal Editor / Asset 설정 영향

Validation
- 확인한 내용
- 아직 확인이 필요한 내용

Next Step
- 권장 다음 작업
```

---

## 질문이 필요한 경우

다음 경우에는 사용자에게 질문합니다.

- 요구사항이 서로 충돌할 때
- 서버 권한 구조가 여러 방식으로 갈릴 때
- 에셋 구조를 알 수 없어 구현이 막힐 때
- 프로젝트 범위를 넓히는 선택이 필요할 때

단, 사소한 구현 선택은 합리적으로 판단하고 진행합니다.
