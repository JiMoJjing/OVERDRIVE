# Scope Control Rules

## 기본 원칙

넓고 미완성인 시스템보다 작고 동작하는 Vertical Slice를 우선합니다.

이 프로젝트는 포트폴리오용이므로, 기능 수보다 설명 가능한 핵심 시스템 완성도가 중요합니다.

---

## 현재 첫 번째 마일스톤

```text
Dedicated Server 로비
→ 플레이어 Ready 상태
→ 메카 파츠 선택
→ Replication 되는 파츠 장착
→ 파츠에 따른 GAS Ability Grant / Remove
```

---

## 명시적 요청 없이 추가하지 않을 것

- 로그라이크 업그레이드 시스템
- 오픈월드 맵
- PvP 시스템
- 대형 인벤토리 시스템
- 상점 / 과금 / 메타 성장 시스템
- 과도한 파츠 종류
- 완성형 UI 연출
- 대량의 적 / 보스 콘텐츠
- 스토리 시스템
- 퀘스트 시스템

---

## Placeholder 허용 기준

다음 경우 Placeholder를 사용할 수 있습니다.

- 시스템 검증이 목적일 때
- 에셋이 아직 확정되지 않았을 때
- 네트워크 / GAS 구조 검증이 우선일 때
- 포트폴리오 설명에 핵심이 아닌 시각 요소일 때

---

## 확장 판단 기준

기능 확장은 다음 질문을 통과해야 합니다.

1. 현재 마일스톤에 필요한가?
2. 포트폴리오에서 기술적으로 설명할 가치가 있는가?
3. Dedicated Server / GAS / UE C++ 역량 증명에 도움이 되는가?
4. 구현 비용 대비 효과가 있는가?
5. 지금 하지 않으면 다음 단계가 막히는가?
