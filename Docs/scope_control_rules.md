# Scope Control Rules

## 기본 원칙

넓고 미완성인 시스템보다 작고 동작하는 Vertical Slice를 우선합니다.

이 프로젝트는 포트폴리오용이므로, 기능 수보다 설명 가능한 핵심 시스템 완성도가 중요합니다.

---

## 현재 첫 번째 마일스톤

```text
Dedicated Server 로비
→ 플레이어 Ready 상태
→ 기본 인간형 플레이어 적용
→ 기본 총기 발사 Ability
→ 적 Placeholder 처치
→ 서버 권한 총기 파츠 드랍
→ 서버 권한 파츠 획득 / 장착 / 교체
→ 파츠에 따른 GAS Ability / GameplayEffect 변화
→ 무기 상태 Replication
```

---

## 명시적 요청 없이 추가하지 않을 것

- 복잡한 로그라이크 업그레이드 시스템
- 오픈월드 맵
- PvP 시스템
- 대형 인벤토리 시스템
- 상점 / 과금 / 메타 성장 시스템
- 과도한 파츠 종류
- 파츠 등급 / 랜덤 옵션 / 시너지 빌드 시스템
- 캐릭터 외형 커스터마이징
- 완성형 UI 연출
- 대량의 적 / 보스 콘텐츠
- 스토리 시스템
- 퀘스트 시스템

---

## 허용되는 제한적 성장 구조

본 프로젝트의 인게임 성장은 로그라이크 레벨업이 아니라, 미션 중 획득한 총기 파츠를 통해 무기를 조립하고 성능을 변화시키는 제한적 성장 구조로 설계합니다.

초기 구조는 다음을 우선합니다.

```text
드랍된 파츠와 상호작용
→ 서버가 획득 가능 여부 검증
→ 현재 무기의 해당 슬롯 파츠와 즉시 교체
→ 기존 파츠는 제거 또는 드랍 처리
→ 변경된 무기 상태 Replication
```

---

## Placeholder 허용 기준

다음 경우 Placeholder를 사용할 수 있습니다.

- 시스템 검증이 목적일 때
- 에셋이 아직 확정되지 않았을 때
- 네트워크 / GAS 구조 검증이 우선일 때
- 포트폴리오 설명에 핵심이 아닌 시각 요소일 때
- 적 / 보스 / 맵보다 총기 파츠 구조 검증이 우선일 때

---

## 확장 판단 기준

기능 확장은 다음 질문을 통과해야 합니다.

1. 현재 마일스톤에 필요한가?
2. 포트폴리오에서 기술적으로 설명할 가치가 있는가?
3. Dedicated Server / GAS / UE C++ 역량 증명에 도움이 되는가?
4. 구현 비용 대비 효과가 있는가?
5. 지금 하지 않으면 다음 단계가 막히는가?
