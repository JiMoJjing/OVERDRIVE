# Unreal Development Rules

## 기본 원칙

이 프로젝트는 Unreal Engine C++ 기반 포트폴리오 프로젝트입니다.

구현은 다음 기준을 따릅니다.

1. 서버 권한 우선
2. Unreal 클래스 책임 준수
3. GAS와 총기 파츠 시스템의 연결성 유지
4. 설명 가능한 구조
5. 작은 Vertical Slice 우선

---

## Unreal 클래스 책임

### GameMode

- 서버 전용 게임 흐름 관리
- 미션 시작 / 종료 판단
- 승패 조건 처리
- 파츠 드랍 생성 규칙 처리
- 서버 권한의 매치 규칙 처리

클라이언트가 알아야 하는 상태를 GameMode에만 저장하지 않습니다.

### GameState

- 클라이언트가 알아야 하는 매치 상태 저장
- 현재 미션 상태
- 보스전 진행 상태
- 결과 화면에 필요한 공용 정보
- 필요 시 월드 파츠 드랍 요약 상태

### PlayerState

- 플레이어 단위의 Replicated 상태 저장
- Ready 상태
- 장착된 무기 파츠 요약 정보
- 킬 / 데미지 / 힐량 / 사망 수 같은 전적 정보
- 결과 화면에 표시할 획득 파츠 또는 기여도 정보

### PlayerController

- 입력 처리
- UI 소유
- 클라이언트에서 서버로 요청 전달
- 로비 UI와 Ready 요청 처리
- 파츠 획득 / 장착 요청 전달

### Character

- Pawn 단위 실제 게임플레이 동작
- 이동
- 피격
- 무기 장착 상태의 시각적 반영
- Hand IK, 조준, 발사 위치 기준 Actor 역할
- Ability 사용의 Avatar 역할

### ActorComponent

- 기능 단위 분리
- 무기 컴포넌트
- 총기 파츠 컴포넌트
- 체력 컴포넌트
- Ability 보조 컴포넌트

## 개발 방향

초기에는 완성형 시스템보다 검증 가능한 Vertical Slice를 우선합니다.

예시:

```text
로비 입장
→ Ready 상태 변경
→ 서버가 Ready 상태 확정
→ 모든 클라이언트에 Replication
→ GameState/UI에서 확인
```

이후 다음 흐름을 연결합니다.

```text
적 처치
→ 서버가 총기 파츠 드랍 생성
→ 클라이언트가 파츠 획득 요청
→ 서버가 요청 검증
→ 무기 파츠 상태 변경
→ GAS Ability / GameplayEffect 갱신
→ Replication으로 클라이언트 갱신
```

## 에셋 사용 원칙

- 시스템 검증 단계에서는 Placeholder 에셋 사용 가능
- 에셋 구조 때문에 핵심 코드가 과도하게 복잡해지지 않게 합니다.
- 플레이어 캐릭터는 단순 인간형 캐릭터를 우선 사용합니다.
- 총기와 총기 파츠 에셋은 런타임 장착, 소켓, 발사 위치 검증에 적합한 구조를 우선합니다.
- 기존 Mech Constructor 기반 로봇 방향은 보류 또는 제외하며, 필요 시 적 / 보스 / 테스트용 에셋으로만 활용합니다.
